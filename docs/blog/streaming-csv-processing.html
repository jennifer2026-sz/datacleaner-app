Streaming CSV Processing: How We Handle 100GB+ Dumps Without OOM

DataCleaner v0.2.0 ships with a complete rewrite of our CSV processing pipeline. The old approach — load entire file into memory, process, write — worked fine for 10MB customer lists. It collapsed catastrophically at 500MB production database dumps. Here's how we fixed it.

The Memory Bomb

Python's csv.DictReader is an iterator — it doesn't load the entire file at once. But our old code did this:

reader = csv.DictReader(f)
rows = list(reader)  # ← BOOM on 500MB file

That list() call forces the entire iterator into memory. A 500MB CSV with 2 million rows × 20 columns ≈ 3-4GB of Python dict objects. Swap death on anything less than a workstation.

The Fix: Single-Iterator Two-Phase Streaming

We replaced the eager load with a streaming architecture that never holds more than 10,000 rows in memory at once:

# Phase 1: read first chunk → classify sensitive columns
chunk_iter = csv_chunked_reader("dump.csv", chunk_size=10000)
headers, first_chunk = next(chunk_iter)
sensitive_cols = classify_columns(first_chunk)

# Phase 2: stream-scrub remaining chunks from SAME iterator
for _, batch in chunk_iter:
    for row in batch:
        for col in sensitive_cols:
            row[col] = deterministic_fake(row[col])
    writer.writerows(batch)

Why a Single Iterator Matters

The naive approach — call csv_chunked_reader() twice — opens the file twice. You'd process the first chunk twice, corrupting row counts. The single-iterator pattern ensures each row is processed exactly once. The first chunk does double duty: classification sample AND scrubbed output. Zero waste.

Text Streaming with Overlap

For document scanning, chunking creates a new problem: what if a social security number straddles the boundary between two chunks? Our text reader carries 200 bytes of overlap:

Chunk N:   "...patient SSN: 123-45-"
Chunk N+1: "123-45-6789 was recorded on..."  ← overlap caught the split!

Results

On a consumer laptop (16GB RAM, RTX 5070 Ti):

  • 10MB CSV: 0.3s (no change — already fast)
  • 100MB CSV: 2.1s (was: 8.4s + 2GB RAM)
  • 500MB CSV: 11s (was: OOM crash)
  • 2GB CSV: 47s (was: impossible)

Memory usage stays flat at ~150MB regardless of file size. The days of "sorry, your database dump is too large" are over.
